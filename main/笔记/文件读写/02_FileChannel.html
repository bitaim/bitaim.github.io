    # FileChannel

      文件通道FileChannel是NIO中引入的，提供了更加丰富的API，支持顺序读写，随机读写。

    # 读写API

    public class NIOFileTest {
    public static void main(String[] args) throws IOException {
        RandomAccessFile randomAccessFile = new RandomAccessFile(new File("D:/file", "record0.txt"), "rw");
        FileChannel fileChannel = randomAccessFile.getChannel();

        // 初始化的FileChannel的position为0
        long initPosition = fileChannel.position();

        fileChannel.write(ByteBuffer.wrap("小鬼".getBytes()));

        // 写入数据后，position被更新为：6
        System.out.println("写入数据后，position被更新为：" + fileChannel.position());

        ByteBuffer byteBuffer = ByteBuffer.allocate(6);

        // 读取不到时，position不会被更新
        fileChannel.read(byteBuffer);

        System.out.println("读取到的数据为：" + new String(byteBuffer.array()));

        // 读取数据后，position被更新为：12
        System.out.println("读取数据后，position被更新为：" + fileChannel.position());

        // 指定position
        fileChannel.position(15);

        // 指定position，position被更新为：15
        System.out.println("指定position，position被更新为：" + fileChannel.position());

        // 这种指定位置的读写，并不会更新position
        fileChannel.write(ByteBuffer.wrap(new byte[1024]), 100);
        fileChannel.read(ByteBuffer.allocate(12), 100);

        // 最终position被更新为：15
        System.out.println("最终position被更新为：" + fileChannel.position());

        // 关闭时，同时会关闭fileChannel
        randomAccessFile.close();
    }
}

    # 源码

      FileChannel的实现类FileChannelImpl。FileChannel的write和read方法均是线程安全的，内部通过一把private final Object positionLock = newObject();锁来控制并发，保证同一时刻只有一个线程可以读写。但是并没有保证顺序读写。部分代码如下。

    public class FileChannelImpl extends FileChannel {
    
    // 当操作 position 时进行加锁
    private final Object positionLock = new Object();
    
    public int write(ByteBuffer src) throws IOException {
        ensureOpen();
        if (!writable)
            throw new NonWritableChannelException();
        // 加锁
        synchronized (positionLock) {
            int n = 0;
            int ti = -1;
            try {
                begin();
                ti = threads.add();
                if (!isOpen())
                    return 0;
                do {
                    n = IOUtil.write(fd, src, -1, nd);
                } while ((n == IOStatus.INTERRUPTED) && isOpen());
                return IOStatus.normalize(n);
            } finally {
                threads.remove(ti);
                end(n > 0);
                assert IOStatus.check(n);
            }
        }
    }
    
}

      FileChannelImpl是通过IOUtil来进行读写的，其中部分代码如下。

    public class IOUtil {

    static int write(FileDescriptor fd, ByteBuffer src, long position,
                     NativeDispatcher nd)
        throws IOException
    {
        // ByteBuffer是堆外内存，直接可写
        if (src instanceof DirectBuffer)
            return writeFromNativeBuffer(fd, src, position, nd);

        // Substitute a native buffer
        int pos = src.position();
        int lim = src.limit();
        assert (pos <= lim);
        int rem = (pos <= lim ? lim - pos : 0);
        // 创建一块临时堆外内存
        ByteBuffer bb = Util.getTemporaryDirectBuffer(rem);
        try {
            // 堆内内存数据拷贝到临时堆外内存
            bb.put(src);
            bb.flip();
            // Do not update src until we see how many bytes were written
            src.position(pos);
            // 基于临时堆外内存进行读写
            int n = writeFromNativeBuffer(fd, bb, position, nd);
            if (n > 0) {
                // now update src
                src.position(pos + n);
            }
            return n;
        } finally {
            Util.offerFirstTemporaryDirectBuffer(bb);
        }
    }

    private static int writeFromNativeBuffer(FileDescriptor fd, ByteBuffer bb,
                                                 long position, NativeDispatcher nd)
            throws IOException
    {
        // ByteBuffer 可读位置
        int pos = bb.position();
        int lim = bb.limit();
        assert (pos <= lim);
        // ByteBuffer 可读容量
        int rem = (pos <= lim ? lim - pos : 0);
    
        int written = 0;
        if (rem == 0)
            return 0;
        if (position != -1) {
            // fd：指定文件
            // ((DirectBuffer)bb).address() + pos：堆外内存的地址
            // rem：数据长度
            // position：文件指针，写到哪个位置
            written = nd.pwrite(fd,
                                ((DirectBuffer)bb).address() + pos,
                                rem, position);
        } else {
            // 堆外内存的一块数据，写到文件描述符的当前指针
            written = nd.write(fd, ((DirectBuffer)bb).address() + pos, rem);
        }
        if (written > 0)
            bb.position(pos + written);
        return written;
    }
}

    # 保证顺序读写

      虽然FileChannel内部已经有了positionLock锁，但并不能保证顺序读写。因为执行的时序可能是不同的，导致文件空洞。所以要在方法外，加一把锁来保证顺序读写。

      顺序读写时，需要对write和read加锁。加锁并不可怕，文件IO操作并没有那么依赖多线程，但是加锁之后的顺序读写必须无法打满磁盘IO。可以通过采用文件分区的方式来达到一举两得的效果，既满足了顺序读写，又减少了锁的冲突。

      那么分多少合适呢？文件多了，锁冲突降级了。文件太多了，又导致碎片化过于严重，单个文件的数据太少，缓存也就不容易命中。就需要benchmark everything。

      当内存吃紧时，page cache的预读分配会受到影响。

    # 堆外内存

      当ByteBuffer申请为堆内内存时，可以看到在读写时都需要创建一块临时的堆外内存。这是因为FileChannel的读写方式，是基于内存地址的。而我们的JVM虚拟机可能在执行读写操作时，发生垃圾回收，而移动内存中对象，可能会导致后续的读写发生错误。当申请一块堆外内存时，JVM就不会影响到堆外内存的回收，因此读写是安全的。

    # 使用技巧

      文件通道 FileChannel 只有在一次性写入4KB的整数倍的数据时，才能保证高性能的写入。这得益于FileChannel采用了ByteBuffer这样的内存缓冲区，可以非常精准的控制写盘的大小，从而减少系统调用次数来提升性能。这是普通IO无法实现的。因此比普通IO要快。

      磁盘的最小写入单位是4KB，如果一次写入小于4KB的数据，实际上耗时和4KB是一样的。可以开辟一块内存缓冲区（类似BufferedInputStream），当缓冲区快满的时候，在进行写文件。

      一次性写入的文件大小，具体还要取决于机器的磁盘结构，并且受到操作系统，文件系统，CPU的影响，有的可能一次性写入64kb才能发挥出最高的IOPS。

      文件通道FileChannel并不是把ByteBuffer中的数据直接写入到磁盘的。ByteBuffer和磁盘中的数据还隔着一层page cache，是用户内存和磁盘之间的一层缓存。可以认为fileChannel.write写入page cache便是完成了落盘操作，但实际上，操作系统最终帮我们完成了page cache到磁盘的最终写入。而FileChannel中提供了一个force方法，用于通知操作系统进行及时的刷盘。

      申请堆外内存时，会校验是否有足够的堆外内存空间，如果空间不足，则会执行System.gc()来尝试回收部分堆外空间，再分配堆外内存。创建堆外内存的消耗要大于堆内内存的消耗，所以当分配了堆外内存之后，尽可能复用它。
