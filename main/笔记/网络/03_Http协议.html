    # HTTP协议

      Http协议是最常用的应用层协议。HTTP协议是建立在TCP/IP协议之上的。

    1、一个Http数据包的结构

    4、数据包是如何传递的？

      Http请求首先封装成Http数据包。然后封装进TCP数据包，再封装进IP数据包，再封装进以太网数据包。

      如果以太网数据包过大，将会被拆分成多个数据包。

      最终数据包传递链路，以太网->交换机广播->网关->N个网关转发->目标机器->再一层一层拆包->解析到Http数据包->由tomcat封装http数据包到HttpServletRequest中->做业务处理->再响应出去->再次封装数据包->按原路径返回。

    5、Http的各个版本

      1、Http1.0，默认是短连接，可以通过指定keep-alive来开启持久化连接。浏览器每次请求都要重新建立一次tcp连接，然后释放连接。

      2、Http1.1，HTTP1.1默认支持长连接，浏览器打开一个网页之后，底层一直保持TCP连接，不会立马断开。之后加载css、js之类的请求，都会基于这个TCP连接。HTTP1.1还支持host头，也就可以支持虚拟主机，而且对断点续传有支持。

      3、Http2.0，HTTP2.0支持多路复用，基于一个tcp连接并行发送多个请求以及接收响应，解决了http1.1对同一时间同一个域名的请求有限制的问题。二进制分帧，将传输数据拆分为更小的帧，提高了性能，实现低延迟高吞吐。

    6、HTTP的长连接是什么？

      HTTP本身没有长、短连接一说，关键在于底层的TCP是什么连接。

      HTTP1.0的底层TCP是短连接，网页发起的一个请求，都要先经过三次握手，发送请求，获取响应，然后四次挥手断开连接。

      HTTP1.1的底层TCP是长连接。三次握手建立连接之后，无论多少请求都是走一个TCP连接的，然后四次挥手断开连接。长连接的方案在业内其实不常用，只是在一些特殊的场景下需要用长连接。

    # TCP协议

      传输控制协议（TCP）是Internet一个重要的传输层协议。TCP提供面向连接、可靠、有序、字节流传输服务，应用程序在使用TCP之前，必须先建立TCP连接。

      1、TCP包头示意图如下：

      源端口号和目标端口号是必不可少的，不然数据就不知道该发给哪个应用了。

      序号是为了解决包乱序的问题。

      确认号是为了确认发出去对方是否有收到，不然会重发直到送达，为了解决不丢包的问题。

      状态位，例如 SYN 是发起一个连接，ACK 是回复，RST 是重新连接，FIN 是结束连接等。TCP 是面向连接的，因而双方要维护连接的状态，这些带状态位的包的发送，会引起双方的状态变更。

      窗口大小，TCP要做流量控制，通信双方各声明一个窗口（缓存大小），标识自己当前能够的处理能力。

    3、如何断开TCP连接？

      这就叫提到大名鼎鼎的面试真题，什么是四次挥手。

      TCP四次挥手断开连接，本质来说是在发送数据之前，先要检测一下网络是否通畅。

    4、为什么TCP建立连接两次不行？

      当客户端第一次与服务端建立连接时，不知道什么原因迟迟建立不了。于是客户端重新发送了第一次握手，服务端响应了第二次握手准备进行通信，并成功返回响应，并关闭了连接。

      后来第一次发送的连接居然到达了服务端，服务端响应第二次握手并开辟了资源准备与客户端进行通信。客户端收到第二次握手之后没有任何表示，客户端认为请求已经结束了，服务端也在等着客户端的请求，浪费了服务端的资源。

      如果说是三次握手，客户端发现第二次握手不是我需要的，因为这个连接之前已经处理过了。客户端遵循三次握手的过程，要求释放资源（复位连接）。

      为什么TCP建立连接四次不行？三次握手已经足够了，更多的握手只是浪费资源。

      总结三次握手，目的是保证双方都有发送和接收的能力。

    5、四次挥手流程

      因为TCP连接是全双工的网络协议，允许通信的双方同时进行数据的收发，同样也允许收发两个方向的连接被独立关闭，以避免client数据发送完毕， 向server发送FIN关闭连接，而server还有发送到client的数据没有发送完毕的情况。 所以关闭TCP连接需要进行四次握手，每次关闭一个方向上的连接需要FIN和ACK两次握手。

      客户端与服务端都处于established状态。

      （1）客户端向服务端发送FIN请求，请求释放连接。随后客户端进入FIN_WAIT_1状态。

      （2）服务端收到客户端的FIN请求，响应ACK给客户端。随后服务端进入CLOSE_WAIT状态。

      （3）客户端收到服务端的ACK响应。随后客户端进入FIN_WAIT_2状态。

      （4）服务端处理完请求之后，向客户端发送FIN请求。随后服务端进入LAST_ACK状态。

      （5）客户端收到服务端的FIN请求，响应ACK给服务端。随后客户端进入TIME_WAIT状态，此时需要等待2MSL。

      （6）服务端收到客户端的ACK响应。随后服务端进入CLOSED状态，关闭了连接。

      MSL是数据包单向传输的最大持续时间。服务端在1MSL内，没有收到客户端的ACK响应，则继续发送FIN请求给客户端。客户端收到请求后，重置TIME_WAIT状态的时间，发送ACK响应给服务端。

      2MSL是为了保证被关闭连接的一方能够正确的关闭连接。

    # UDP协议

      用户数据报协议UDP是Internet传输层协议。提供无连接、不可靠、数据报尽力传输服务。

      数据报文如下图所示。

    1、UDP协议的特点

      应用进程更容易控制发送什么数据以及何时发送。无需建立连接。无连接状态。首部开销小。

      对于数据可靠性要求不高，比如音视频聊天，视频卡顿一下、模糊一下是能够接收的。或者物联网的设备日志上报，偶尔丢一个影响不大。

    2、TCP与UDP的比较

    TCP UDP
    面向连接 无连接
    提供可靠性保证 不可靠
    慢 快
    资源占用多 资源占用少
