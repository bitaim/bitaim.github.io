    # 网络IO模型

      Linux中常用的4种网络IO模式。

    • 阻塞I/O（blocking IO）
    • 非阻塞I/O（nonblocking IO）
    • I/O多路复用（IO multiplexing）
    • 异步I/O（asynchronous IO）

    # 阻塞I/O

      在同步阻塞IO模式种，程序的IO系统调用开始，直到系统调用返回。这段时间内，进程（Linux下线程同样也是进程）都是阻塞的，也就是当前线程被阻塞。阻塞分为两个阶段：

    • 等待数据就绪。对于网络IO而言，是等待远端的数据到达。对于磁盘IO而言，是等待磁盘数据从磁盘读取到内核缓冲区。
    • 复制的复制。等待内核缓冲区的数据复制到用户空间的缓冲区。

      直到成功返回后，进程才开始处理用户空间的缓冲区数据。

    特点

      优点。开发简单。在阻塞等待数据期间，用户线程挂起，不占用CPU资源。

      缺点。需要为每一个连接分配一个线程，在高并发应用场景下，需要大量的线程来维护大量的网络连接，内存、线程切换开销会非常大。

    # 非阻塞I/O

      在内核缓冲区没有数据的情况下，系统调用会立即返回调用失败的结果，线程不会被阻塞。

      用户线程需要不断的发起IO系统调用，看看内核数据是否准备好了没有。

      内核数据准备好了以后，用户线程发起系统调用，此时用户线程被阻塞，开始将内核缓冲区的数据复制到用户缓冲区，然后系统调用返回。

      用户线程开始执行业务逻辑。

    低。

    特点

      优点。每次发起的IO系统调用，在内核等待数据过程中可以立即返回。用户线程不会阻塞，实时性较好。

      缺点。用户进程使用非阻塞I/O需要轮询方式进行系统调用以询问内核是否准备好数据，效率低下。

    # I/O多路复用

      在IO多路复用种，引入了三种新的系统调用select、poll、epoll。可以做到一个线程同时监视多个文件描述符，由内核来判断是否数据到来，一旦某个文件描述符就绪（一般是内核缓冲区可读/可写），内核能够将就绪的状态返回给应用程序。

      应用程序根据就绪的状态，进行相应的IO系统调用。

      使用这些系统调用，就可以用一个线程同时监视成百上千的Socket连接，一旦有Socket网络连接有IO就绪的状态，就返回对应的可执行的读写操作。

    特点

      优点。使用一个线程就可以同时轮询成百上千的连接。系统不必要创建和维护大量的线程，大大减小的系统的开销。

      缺点。select/epoll调用都是阻塞时的，属于同步IO。都需要在读写事件就绪后，由系统调用本身负责读写。也就是这个读写过程是阻塞的。

    1、select

    #include <sys/select.h>

// 最关注的读描述符集 其他传null
int select(int nfds, fd_set *readfds, fd_set *writefds,
           fd_set *exceptfds, struct timeval *timeout);

void FD_CLR(int fd, fd_set *set);
int  FD_ISSET(int fd, fd_set *set);
void FD_SET(int fd, fd_set *set);
void FD_ZERO(fd_set *set);

int pselect(int nfds, fd_set *readfds, fd_set *writefds,
            fd_set *exceptfds, const struct timespec *timeout,
            const sigset_t *sigmask);

      其中fd_set是一个bitmap，由内核参数来控制大小，最大长度为1024，代表可以同时监听1024个文件描述符。虽然可以调整，但性能可能会更差。

      由于bitmap是下从下标0开始的，如果我们需要监听3、4、8这三个文件描述符，就需要传入nfds=9（最大文件描述符+1），这样才可以监听。

    1. 发起select系统调用时，会把fd_set从用户空间复制到内核空间，应用程序进入阻塞状态。
    2. 内核根据fd_set得到需要处理器的文件描述符，如果相应的准备好了数据，就会将fd_set进行置为。随后系统调用返回。
    3. 进程被唤醒后，拿到内核处理过的fd_set，然后通过FD_ISSET判断是否被置位，然后就可以执行相应的后续IO操作了。但这里需要遍历所有的文件描述符，因为不知道哪一个被置位。
    4. 下一次发起select系统调用时，又要重新准备fd_set，因此已经被置位过，没有办法再次使用了。

    2、poll

    #include <poll.h>

// pollfd *fds 数组
// nfds 数量
// 超时时间
int poll(struct pollfd *fds, nfds_t nfds, int timeout);
            
struct pollfd {
  int   fd;         /* 待检测的文件描述符 */
  short events;     /* 监测的事件 */
  short revents;    /* 监测到数据时，给该字段置位 */
};
    1. 发起poll系统调用，会把pollfd从用户空间拷贝到内核空间，应用程序进入阻塞状态。
    2. 内核根据文件描述符的监听事件，给pollfd的revents进行置位。
    3. 进程被唤醒，拿到内核处理后的pollfd，判断对应事件是否被置位，然后执行后续操作。

      优点。

      与select类似，但使用数组取代了bitmap，没有了文件描述符1024的限制。

      引入了pollfd结构体，只需要重置revents字段，就可以做到复用pollfd。

      缺点。

      每次调用poll都需要拷贝新的poll_fd到内核空间，存在用户态到内核态的切换。

      拿到pollfd之后，需要全部遍历，才知道哪些文件描述符被置位。

    3、epoll

      epoll是Linux特有的I/O复用函数，在实现上与select、poll有很大的差异。epoll把用户关心的文件描述符放在内核的一个事件表里，而无须像select、poll那样每次调用都需要传入文件描述符或集合集。但epoll需要一个额外的文件描述符，来唯一标识内核中的一个事件表。

      epoll的事件表是一颗红黑树。

    1. 使用int epoll_create(int size)创建一个事件表，并返回一个文件描述符。
    2. 使用epoll_ctl向事件表中添加、修改、删除需要监听的文件描述符及事件。
    3. 使用epoll_wait系统调用等待内核的监听结果，进程开始阻塞。内核会将准备好的文件描述符放入就绪队列中。然后返回该就绪队列。
