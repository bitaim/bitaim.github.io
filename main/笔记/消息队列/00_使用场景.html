# 使用场景

  在微服务场景下，服务之间互相通信调用通常采用基于OpenFeign的Http方式调用或者基于Dubbo的RPC方式调用，这两种方式都是直接调用的方式。

  而消息队列，是服务之间通信的第三种方式。对于消息队列，我们通常会归纳为三种使用场景，分别是解耦、异步、削峰。

  1、解耦

  2、异步

  对于业务中某些非核心的功能，可以采用异步的方式来提高性能。

  比如新人创建账号后，需要下发一些优惠卷，或者发送短信提醒，这些功能均可以采用异步的方式去执行。这些功能均不是核心功能，即使发送失败，可以通过人工补足。

  或者你会说，在业务中使用线程池也可以做到异步的方式，那么为什么还需要使用消息队列呢？

  （1）使用线程池的方式，无疑会增加系统的性能损耗。

  （2）对于一个优秀的分布式系统，肯定已经有了成熟的短信系统。我们无须在意短信是如何发送的，只需要关心接收人和短信内容（短信模板ID，参数）即可。只需要下发消息到消息队列即可，大大降低开发成本。

# 消息队列协议

1、AMQP协议

  AMQP协议是一套标准的底层协议，直接定义了网络交换的数据格式，并且兼容了JMS。基于该协议的消息队列产品比较多，最熟悉的就是RabbitMQ。

  由于基于同一协议，其最大的特点就是跨语言（使用Java语言发送消息，使用go语言消费消息）。

2、MQTT协议

  MQTT是消息队列遥测传输，是专门为小设备而设计的协议。适用于计算性能不高的设备，主要在物联网（IOT）中大量的使用。

  其特征是，内存占用低，为小型设备之间通过低带宽发送短消息而设计。支持主题发布和订阅。

  不支持长周期存储和转发，简单用户名和密码，不支持安全连接，消息不透明。

# 消息队列选型

  1、kafka

  Kafka由Scala和Java编写，是一种高吞吐量的分布式发布订阅消息系统（严格意义上不属于队列产品，是一个流处理平台），它可以处理大规模网站中的所有动作流数据（网页浏览、搜索和其他用户的行动），副本集机制，实现数据冗余，保证数据尽量不丢失。

  特点，类似MQ，支持简单的MQ功能。运维难度大，文档比较少，需要掌握Scala。

  2、RocketMQ

  阿里开源的消息队列产品，纯Java开发，具有高吞吐量、高可用性、适合大规模的分布式系统应用。性能强劲，支持海量堆积，支持指定次数和时间间隔的失败消息重发，支持consumer端tag过滤、延迟消息等。在阿里内部进行大规模使用，适合电商、互联网金融等领域。

  3、RabbitMQ

  是一个开源的AMQP实现，使用Erlang语言编写，适合各类平台的客户端。
