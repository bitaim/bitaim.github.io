    # 设计原则

    # 单一职责原则

      单一职责原则的定义是，一个类只负责完成一个职责或者功能。也就是说，不要设计大而全的类，要设计粒度小、功能单一的类。一个类如果包含了两个或者两个以上业务不相干的功能，那就说明它的职责不够单一，应该将它拆分成多个功能更加单一、粒度更细的类。

      比如一个类里既包含订单的一些操作，又包含用户的一些操作。而订单和用户是两个独立的业务领域模型，将两个不相干的功能放到同一个类中，那就违反了单一职责原则。需要将这两个类拆分为两个粒度更细、功能更加单一的两个类：订单类和用户类。

      单一职责原则在真实情况下，不是那么容易区分的。比如下面的这个案例。

    public class UserInfo {  
    private long userId;  
    private String username;  
    private String email;  
    private String telephone;  
    private long createTime;  
    private long lastLoginTime;  
    private String avatarUrl;  
    private String provinceOfAddress;       // 省  
    private String cityOfAddress;           // 市  
    private String regionOfAddress;         // 区
    private String detailedAddress;         // 详细地址

      一种观点是，UserInfo类包含的都是跟用户相关的信息，满足单一职责原则。另一种观点是，地址信息在UserInfo类中，所占的比重比较高，可以继续拆分成独立的UserAddress类，UserInfo类中只保留除地址之外的其他信息，拆分之后的两个类的职责更加单一。

      是否符合单一职责原则还要根据具体的应用场景而定。如果在社交产品中，用户的地址信息与其他信息一样，只是单纯的用来做展示，那么UserInfo类现在的设计就是合理的。

      但是，如果这个社交产品发展得比较好，之后又在产品中添加了电商的模块，那么用户的地址信息还会用在电商物流中，那么我们最好将地址信息从UserInfo类中拆分出来，独立成用户物流信息（或者叫地址信息、收货信息等）。

      如果这个社交产品发展得越来越好，公司内部又开发出了更多产品。公司希望支持统一账号系统，也就是用户一个账号可以在公司内部的所有产品中登录。这个时候，就需要对UserInfo类进行拆分，将跟身份认证相关的信息（比如email、telephone等）抽取成独立的类。

      单一职责原则不是越单一越好，一切都要向着可读性、可维护性方向靠齐，需要自己掌握好这个度。

    # 开闭原则

      开闭原则指的是对扩展开放，对修改关闭。开闭原则是最为重要的设计原则。因为扩展性是代码质量最重要的衡量标准。对拓展开放是为了应对变化（需求），对修改关闭是为了保证已有代码的稳定性，最终结果是为了让系统更有弹性。

      当添加一个新的功能时，应该在已有代码上拓展代码，而非修改已有代码。

    # 里氏替换原则

      里氏替换原则指的是子类对象能够替换程序中父类对象出现的任何地方，并且保证原来程序的逻辑行为不变及正确性不被破坏。

    public class Transporter {

    private HttpClient httpClient;

    public Transporter(HttpClient httpClient) {
        this.httpClient = httpClient;
    }

    public Response sendRequest(Request request) {
        // send request
    }

}

    public class SecurityTransporter extends Transporter{

    private String appId;
    private String appToken;

    private HttpClient httpClient;

    public Transporter(HttpClient httpClient, String appId, String appToken) {
        this.httpClient = httpClient;
        this.appId = appId;
        this.appToken = appToken;
    }

    public Response sendRequest(Request request) {
        if (StringUtils.isNotBlank(appId) && StringUtils.isNotBlank(appToken)) {
            request.addPayload("app-id", appId);
            request.addPayload("app-token", appToken);
        }
        return super.sendRequest(request);
    }

}

    public class Demo {

    public void demoFunction(Transporter transporter) {
        Request request = new Request();
        Response response = transporter.sendRequest(request);
    }

    public static void main(String[] args) {
        // 里氏替换原则
        Demo demo = new Demo();
        demo.demoFunction(new SecurityTransporter(httpclient, '23', '123'));
    }

}

    2、违背里氏替换原则的行为

      （1）子类违背父类声明要实现的功能。父类中提供的sortOrdersByAmount()订单排序函数，是按照金额从小到大给订单排序的。而子类重写sortOrdersByAmount()订单排序函数之后，是按照创建日期来给订单排序的。那子类的设计就违背里氏替换原则。

      （2）子类违背父类对输入、输出、异常的约定。在父类中，某个函数约定，在运行出错的时候返回null；获取数据为空时返回空集合。而子类重载函数后，实现变了，运行出错返回异常，获取不到数据返回null。那子类的设计就违背了里氏替换原则。

      在父类中，某个函数输入数据可以是任意整数。但子类实现后，只允许输入正整数，负数就抛出异常，子类对输入的数据校验更加严格，那么子类的设计就违背了里氏替换原则。

      在父类中，某个函数只会抛出ArgumentNullException异常，那子类的设计实现中也只允许抛出ArgumentNullException异常，任何其他异常的抛出，都会导致子类违背里氏替换原则。

      （3）子类违背父类注释中所罗列的任何特殊说明。在父类中，用户的提现金额不得超过账户余额。但在子类中，针对VIP账号实现了透支提现的功能。那么子类的设计就违背了里氏替换原则。

    3、与多态的区别

      从定义描述和代码实现上来看，多态和里氏替换优点类似。但它们关注的角度不一样，多态是面向对象编程的一大特性，是一种代码实现的思路。而里氏替换是一种设计原则，用来指导继承关系中子类该如何设计，子类的设计要保证在替换父类的时候，不改变原有程序的逻辑及不破坏原有程序的正确性。

    # 接口隔离原则

      比如一个用户相关的接口，有注册、登录、获取用户信息等。

    public interface UserService {

    void register(String phone, String password);

    void login(String phone, String password);

    UserInfo getUserInfoByPhone(String phone);

    // void deleteByPhone(String phone);

}

      现在后台管理系统要实现删除用户的功能，希望用户希望提供一个删除用户的接口。通常的话，我们就会在UserService接口类中直接定义一个删除函数。虽然可以解决问题，但是也隐藏了一些安全隐患。

      删除用户是一个非常慎重的操作，我们只希望通过后台管理系统来运行。所以这个接口仅限于给后台管理系统使用。如果我们把它放到UserService接口类中，那么所有使用到该类的系统，都可以调用这个接口。不加限制地被其他业务系统调用，就有可能导致误删用户。

      当然最好的解决方案是从架构设计的层面，通过接口鉴权的方式来限制接口的使用。不过，如果暂时没有鉴权框架来支持，我们还可以从代码设计的层面，尽量避免接口被误用。参照接口隔离原则，调用者不应该强迫依赖它不需要的接口，将删除接口单独放到另外一个接口中，然后将该接口单独打包提供给后台管理系统来使用。

      比如一个统计函数，可能返回了多种统计指标，平均值、求和、最大值等。如果我们每次使用该函数，只是为了获取里面的几个指标，那么该函数就做了一些无用功，因此就应该把该函数拆分成更加细粒度的多个函数。
