    # 线程池

    # 创建线程池

    @Configuration
public class ThreadPoolCongfiguration{

    @Bean
    public ThreadPoolExecutor searchExecutor() {
        ThreadPoolExecutor threadPoolExecutor = 
            new ThreadPoolExecutor(5, 10, 5, TimeUnit.SECONDS, new LinkedBlockingQueue<>(5));
        return threadPoolExecutor;
    }

    @Bean
    public ScheduledThreadPoolExecutor scheduledThreadPoolExecutor() {
        // 使用的队列是 DelayedWorkQueue
        return new ScheduledThreadPoolExecutor(5);
    }
    
}

    # 关闭线程池

      （1）调用shutdown()方法。后续提交的任务将会被拒绝，只可以处理正在执行和队列中的任务。

      （2）调用List<Runnable> shutdownNow = threadPoolExecutor.shutdownNow()方法后，中断所有的工作线程，清空队列中的任务并以集合的形式返回。

    # 异常宕机

      如果线上机器突然宕机，线程池的阻塞队列中的请求必然会丢失。在线程池提交任务之前，可以将任务信息保存到数据库，更新请求的状态：未提交、已提交、已完成。提交成功之后，更新状态为已提交。

      系统重启后，后台线程去扫描数据库里的未提交和已提交状态的任务，可以把任务的信息读取出来，重新提交到线程池里去，继续执行。

    # 其他

    1、使用线程池的好处

      线程池，方便对线程的管理。对线程资源进行统一的分配和监控，尽可能地复用线程，而不是频繁创建和销毁线程（降低创建和消耗的性能损耗）。

    2、线程并不是越多越好

      线程占内存。Java中万物皆对象，线程也不例外。既然是对象就需要占用内存。在JVM规范中，一个线程默认最大栈大小为1MB。线程越多，消耗内存越多。

      线程加大操作系统压力。线程不宜过多，操作系统需要频繁切换上下文，影响性能，最终导致所有线程执行都很慢。

    3、如何确定合适数量的线程

      如果是计算型、纯内存的操作，占用CPU会比较高，属于CPU密集型。建议是CPU数量的1~2倍。计算型任务通常采用大队列和小池，可以最大程度地减少CPU使用率，操作系统资源和上下文切换，但会导致人为地降低吞吐量。

      如果是IO型任务，比如调用第三方接口，读写文件等。可以多指定一些线程，需根据具体的IO阻塞时长进行考量决定。IO型任务通常采用小队列和大池，则操作系统的线程调度开销则会增大，同样也会降低吞吐量。

      通过监控CPU情况，来动态的调整线程资源，如果CPU使用率达到80%，那么是充分使用了CPU的资源。调整没有一定的模板，要不断的测试找到一个合适点。
