# 线程

  概念，线程是操作系统能够进行运算调度的最小单位。由操作系统的任务调度系统统一调度安排，为每一个线程分配一定的时间片执行任务。

  对于同一个CPU核心来看，由于分配到线程的时间片时间极短，从宏观上看各个程序是并行任务进行的，从微观上看各个程序是串行执行任务的。在多核CPU的情况下，线程是并行执行任务的。

  对于目前的CPU来说，又出现了4核心8线程的超线程模式。做到4个CPU核心的情况下，最多可以同时运行8个线程的情况。这就是科技的力量。

1、为什么要使用多线程

  使用多线程可以充分使用（压榨）CPU的运算能力，来提升系统的性能。

  在目前的开发工作中，会出现大量磁盘IO和网络IO的操作，而线程在执行这些IO操作时是不会占用CPU资源的。当CPU在执行IO操作时，就会处于空闲状态，所以可以在此时通过多线程的方式来压榨CPU的运算能力。

  当线程执行IO操作时，CPU并不会参与，而是向DMA（Direct Memory Access直接内存访问）控制器下发命令，由DMA去执行相关的IO操作，最终DMA通过中断的形式通知CPU代表已完成IO操作。比如查询数据库、调用第三方接口、读写文件等都不消耗CPU资源，CPU仅仅只是等待响应而已。详情请具体了解DMA，我认为作为开发人员了解到这个程序已经足够了。

2、Java有几种创建线程的方式

  这个问题是经常被问到的，答案多种多样。我们可以在JDK源码中的一段注释中找到这个问题的答案。

 * There are two ways to create a new thread of execution. One is to
 * declare a class to be a subclass of <code>Thread</code>. This
 * subclass should override the <code>run</code> method of class
 * <code>Thread</code>. An instance of the subclass can then be
 * allocated and started. 

 * The other way to create a thread is to declare a class that
 * implements the <code>Runnable</code> interface. That class then
 * implements the <code>run</code> method. An instance of the class can
 * then be allocated, passed as an argument when creating
 * <code>Thread</code>, and started.

  第一种是继承Thread类，重写run()方法。第二种是实现Runnable接口，重写run()方式。仅仅只有这两种方式。其他答案的底层方式也是这两种而已。

3、多线程的合理使用

  频繁创建和销毁线程是有性能损耗的。所以通常我们不会直接通过Thread类直接创建线程，而不是通过自定义线程池来管理线程。

  另外操作系统对线程之间的切换，需要保存上一个线程执行的进度（上下文），是有性能消耗的。因此线程不宜过多，合理的使用线程资源是很有必要的。

# 线程

1、创建线程的方式

  上面提到的两种创建线程的方式，来进行一下优势劣势对比。通过实现Runnable接口的方式来创建线程是最好的方式。

  1. Runnable子类的方式，实现了解耦。
  2. Thread子类的方式，由于Java不支持多继承，限制了拓展性。
  3. Thread子类的方式，只能通过new的方式来创建线程。Runnable子类的方式，可以配合线程池使用，做到复用线程。

2、线程的其他属性

// 设置线程为守护态，创建线程时默认为非守护态。
// 当我们的JVM进程所有的非守护态线程已经停止，那么这些守护态线程也都会停止，JVM进行就会停止
thread.setDaemon(false);
// 设置线程名称
thread.setName("线程名称");
// 设置线程优先级，优先级越大被执行的机会越多
thread.setPriority(10);

3、捕获线程的异常

  未捕获异常的线程，在执行过程中发生异常会直接打印出堆栈信息。

  可以通过指定线程的未捕获异常处理器，来对异常的情况执行适当的操作。下面是示例代码。

public class ThreadExceptionTest {

    public static void main(String[] args) throws InterruptedException {
        caughtException();
    }

    /**
     * 未捕获异常
     *
     * @throws InterruptedException
     */
    private static void unCaughtException() throws InterruptedException {
        Thread thread = new Thread(() -> {
            int number = 10 / 0;
            System.out.println(number);
        });
        thread.start();
        thread.join();
    }

    /**
     * 捕获异常
     *
     * @throws InterruptedException
     */
    private static void caughtException() throws InterruptedException {
        Thread thread = new Thread(() -> {
            int number = 10 / 0;
            System.out.println(number);
        });

        thread.setUncaughtExceptionHandler(new Thread.UncaughtExceptionHandler() {
            @Override
            public void uncaughtException(Thread t, Throwable e) {
                if (e != null) {
                    String threadName = t.getName();
                    System.out.println("线程名：[ " + threadName + " ]运行过程中发生异常，原因：" + e.getMessage());
                }
            }
        });

        thread.start();
        thread.join();
    }

}

4、sleep

/**
* Causes the currently executing thread to sleep (temporarily cease
* execution) for the specified number of milliseconds, subject to
* the precision and accuracy of system timers and schedulers. The thread
* does not lose ownership of any monitors.
* 使当前执行的线程在指定的毫秒数内休眠（暂时停止执行）
* 这取决于系统计时器和调度程序的精度和准确性
* 并不会失去对任何监视器(monitors)的所有权
* 也就是说不会释放锁，仅仅会让出cpu的执行权
*/
public static native void sleep(long millis) throws InterruptedException;

  该方法可以使线程挂起（让出CPU）指定的一段时间，在挂起期间不会释放任何锁。在挂起时间结束后，重新等待操作系统的调度，因此并不会在挂起时间结束后立即执行。

  可以通过Thread.sleep(0)触发操作系统立刻重新进行一次CPU竞争，竞争的结果可能仍然是当前线程获得CPU时间片，也许是其他线程获得CPU时间片。与yield()方法功能相同。

# 停止线程

1、interrupt中断

  通过interrupt()方法来中断（不是停止）线程，将线程的中断状态置为true。需要在定义任务时，通过手写代码去监听（检测）中断标志位做到适时停止线程。

  通常中断方法会用于在一个后台线程，比如这个线程在做一些周期性质的任务，会一直伴随着整个JVM的生命周期存活。在JVM结束时，对这些线程进行中断，做一些停止时的收尾工作。

public class InterruptTest {

    public static void main(String[] args) throws InterruptedException {
        Thread thread = new Thread(new PrintDateTask());
        thread.start();

        // main线程休眠20ms后主动中断子线程
        Thread.sleep(20);
        thread.interrupt();
        // 等待子线程任务结束
        thread.join();
    }

    private static class PrintDateTask implements Runnable {

        @Override
        public void run() {
            // 检测线程是否被中断
            while (!Thread.currentThread().isInterrupted()) {
                System.out.println("当前时间：" + new Date());
            }
            System.out.println("任务被中断，提前结束");
        }

    }

}

2、中断状态

  （1）如果线程正在调用一些阻塞方法时，此时线程被中断（置为true），这些线程会直接响应（检测到）中断状态，抛出InterruptedException异常。这些阻塞方法直接被停止，做到响应中断，同时清除目标线程的中断标志位（置为false）。比如Thread#sleep(long)、Object#wait()等阻塞方法。

  （2）如果线程是被I/O或者NIO中的Channel所阻塞，同样I/O操作会被中断或者返回特殊异常值，达到终止线程的目的。

  （3）如果线程不满足挂起、等待、阻塞或者I/O所阻塞等条件的话，那么会将线程的状态设为中断状态。就需要通过手写代码去检测中断，再去做业务处理了。

3、废弃的stop()方式

  这种方式可能会导致线程的安全问题，造成数据不一致的情况。

public class UnsafeApiStopTest {

    private static int a = 100;
    private static int b = 100;

    public static void main(String[] args) throws InterruptedException {
        Thread thread = new Thread(() -> {
            a = a + 100;
            try {
                Thread.sleep(1000);
            } catch (InterruptedException e) {
            }
            b = b + 100;
        });
        thread.start();
        Thread.sleep(500);
        thread.stop();
        while (thread.isAlive()) {
        }
        // 对于a、b变量来说，要么全部成功，要么全部失败
        // 使用 stop的方式 造成了数据不一致的情况
        // 输出结果：   a：200,b：100
        System.out.println("a：" + a + ",b：" + b);
    }
}

# 生命周期

  （1）NEW，新生状态。创建了线程（new），但还未启动（start）。

  （2）RUNNABLE，运行状态。启动之后的线程状态，但需要等待操作系统的任务调度。当执行阻塞式IO操作时，线程同样处于RUNNABLE状态（这里的状态是JVM中对线程的封装，而在底层操作系统中是WAITING状态）。

  （3）BLOCKED，阻塞状态。有三种情况，

  （3.1）当由于获取不到监视器锁（synchronized），无法进入一个同步代码块或者同步方法时，就会处于阻塞状态。

  （3.2）进入了同步代码块或者同步方法后，在其中调用了wait()方法，释放了监视器锁。当线程被唤醒后，如果竞争不到监视器锁，同样会进入阻塞状态。

  （3.3）当JVM发生垃圾回收时，所有的工作线程都会处于阻塞状态。

  （4）WAITING。当调用Object#wait()、Thread#join()、LockSupport#park()时，就会进入等待状态。

  （5）TIMED_WAITING。当调用Thread#sleep(long)、Object#wait(long)、Thread#join(long)、LockSupport#parkNanos(long)或LockSupport#parkUntil(long)时，就会进入定时等待状态。

  （6）TERMINATED。线程正常或者异常结束，均会进入结束状态。

# 线程通信

  指多个线程在运行期间，线程之间的交互协作。

1、suspend与resume

  这是两个被废弃的API。suspend是挂起目标线程，但不会释放锁。resume来恢复目前线程。使用时必须保证suspend一定要比resume先执行，否则就无法被唤醒了，导致死锁的发生。

2、wait与notify

  方法wait()与方法notify()一定要在同步代码块中使用。虽然wait()会自动释放锁锁，但是对顺序有要求。如果notify()先于wait()使用，那么会造成死锁的发生。

public class NotifyTest {

    private static int[] numberArray = {1, 2, 3, 4, 5};
    private static char[] charArray = {'A', 'B', 'C', 'D', 'E'};
    private static final byte[] lock = new byte[0];
    private static CountDownLatch countDownLatch = new CountDownLatch(1);


    public static void main(String[] args) throws InterruptedException {

        Thread thread1 = new Thread(new PrintNumber());
        Thread thread2 = new Thread(new PrintChar());
        thread2.start();
        thread1.start();


    }

    private static class PrintNumber implements Runnable {

        @SneakyThrows
        @Override
        public void run() {
            // 保证优先打印数字
            countDownLatch.countDown();
            synchronized (lock) {
                for (int number : numberArray) {
                    System.out.println("数字：" + number);
                    // 唤醒其他线程
                    lock.notify();
                    // 阻塞当前线程
                    lock.wait();
                }
                lock.notify();
            }
        }
    }

    private static class PrintChar implements Runnable {

        @SneakyThrows
        @Override
        public void run() {
            countDownLatch.await();
            synchronized (lock) {
                for (char chars : charArray) {
                    System.out.println("字母：" + chars);
                    // 唤醒其他线程
                    lock.notify();
                    // 阻塞当前线程
                    lock.wait();
                }
            }
        }

    }

}

3、park与unpark

  线程调用park则等待许可，unpark方法为指定线程提供许可。不要求park和unpark方法的调用顺序。多次调用unpark之后，在调用park，线程会直接运行。但不会叠加，也就是说，连续多次调用park方法，第一次会拿到许可直接运行，后续调用会进入等待。park和unpark不会对锁进行释放。

public class NotifyTest {

    public class NotifyTest {

    private static int[] numberArray = {1, 2, 3, 4, 5};
    private static char[] charArray = {'A', 'B', 'C', 'D', 'E'};
    private static Thread thread1;
    private static Thread thread2;


    public static void main(String[] args) throws InterruptedException {

        thread1 = new Thread(new PrintNumber());
        thread2 = new Thread(new PrintChar());

        thread1.start();
        thread2.start();

    }

    private static class PrintNumber implements Runnable {

        @Override
        public void run() {
            for (int number : numberArray) {
                System.out.println("数字：" + number);
                LockSupport.unpark(thread2);
                LockSupport.park();
            }
        }

    }

    private static class PrintChar implements Runnable {

        @Override
        public void run() {
            for (char chars : charArray) {
                LockSupport.park();
                System.out.println("字母：" + chars);
                LockSupport.unpark(thread1);
            }
        }
    }

}

4、Condition

public class NotifyTest {

    private static int[] numberArray = {1, 2, 3, 4, 5};
    private static char[] charArray = {'A', 'B', 'C', 'D', 'E'};
    private static ReentrantLock reentrantLock = new ReentrantLock();
    private static Condition numberCondition = reentrantLock.newCondition();
    private static Condition charCondition = reentrantLock.newCondition();


    public static void main(String[] args) throws InterruptedException {

        Thread thread1 = new Thread(new PrintNumber());
        Thread thread2 = new Thread(new PrintChar());
        thread1.start();
        thread2.start();

    }

    private static class PrintNumber implements Runnable {

        @Override
        public void run() {
            reentrantLock.lock();
            try {
                for (int number : numberArray) {
                    System.out.println("数字：" + number);
                    charCondition.signal();
                    numberCondition.await();
                }
                charCondition.signal();
            } catch (InterruptedException e) {
                e.printStackTrace();
            } finally {
                reentrantLock.unlock();
            }
        }

    }

    private static class PrintChar implements Runnable {

        @Override
        public void run() {
            reentrantLock.lock();
            try {
                for (char chars : charArray) {
                    System.out.println("字母：" + chars);
                    numberCondition.signal();
                    charCondition.await();
                }
                numberCondition.signal();
            } catch (InterruptedException e) {
                e.printStackTrace();
            } finally {
                reentrantLock.unlock();
            }
        }
    }
}

# 伪唤醒

  官方建议应该在循环中检查等待条件，原因是处于等待状态的线程可能会收到错误警报和伪唤醒。如果不在循环中检查等待条件，程序就会在没有满足结束条件的情况下退出。涉及到底层的CPU调度。不要使用if判断是否进入等待状态。

  伪唤醒是指线程并非因为notify、notifyAll、unpark等api调用而唤醒，是更底层原因导致的。

synchronized(obj) {
    while(条件) {
        obj.wait();
    }
}

while(条件) {
    LockSupport.park();
}
