# 线程

  线程是程序执行的基本单位。操作系统的任务调度系统，会为每一个线程分配一定的CPU时间片去执行，由于时间极短，在宏观上来看各个程序是同时进行的，在围观上来看各个程序是串行执行的。

1、为什么要使用多线程？

  单线程与多线程的出现主要是由于IO（磁盘IO和网络IO），而对于IO操作其耗时是比较长的。当线程执行IO操作时，CPU并不会参与，而是向DMA（Direct Memory Access直接内存访问）控制器下发命令，由DMA去执行相关的IO操作，最终DMA通过中断的形式通知CPU代表已完成IO操作。比如查询数据库、调用第三方接口、读写文件等都不消耗CPU资源，CPU仅仅只是等待响应而已。

  在现代系统中，IO操作是非常多的，当CPU执行IO操作时，就会处于空闲状态。现代计算机和服务器的CPU都是以多核心为主，使用多线程可以充分使用（压榨）CPU，来提升系统的性能。

2、多线程的合理使用

  频繁创建和销毁线程是有性能损耗的。

  另外线程之间切换，需要保存上一个线程执行的进度（上下文），是有性能消耗的。因此线程不宜过多，合理的使用线程资源是很有必要的。

# 创建线程

1、创建线程的方式

  在Thread的注释中提到，创建线程的方式只有两种。一种是继承Thread类，另一种是实现Runnable接口，这两种方式都要重写run()方法。

  我们所说的线程，在Java中的抽象也就是Thread类，那么创建线程就是如何创建Thread类的实例对象。也就是下面这两种构造方式而已。

public Thread() {
    init(null, null, "Thread-" + nextThreadNum(), 0);
}

public Thread(Runnable target) {
    init(null, target, "Thread-" + nextThreadNum(), 0);
}

2、优缺点对比

  通过实现Runnable接口的方式来创建线程是最好的方式。

  （1）Runnable子类的方式，实现了解耦。

  （2）Thread子类的方式，由于Java不支持多继承，限制了拓展性。

  （3）Thread子类的方式，只能通过new的方式来创建线程。Runnable子类的方式，可以配合线程池使用，做到复用线程。

3、线程的其他属性

  线程另外还有优先级，守护态等属性。

thread.setDaemon(false);
thread.setName("aaa");
thread.setPriority(10);

4、捕获线程中的异常

  未捕获异常的线程，在执行过程中发生异常会直接打印出堆栈信息。

public class CaughtThreadException {

    public static void main(String[] args) throws InterruptedException {

        Thread thread = new Thread(new CalcTask(null, 2));
        // 捕获异常
        thread.setUncaughtExceptionHandler(new Thread.UncaughtExceptionHandler() {
            @Override
            public void uncaughtException(Thread t, Throwable e) {
                System.out.println("异常：" + e.getMessage());
            }
        });
        thread.start();

        thread.join();

        System.out.println("结束");

    }

    private static class CalcTask implements Runnable {

        private Integer num1;

        private Integer num2;

        public CalcTask(Integer num1, Integer num2) {
            this.num1 = num1;
            this.num2 = num2;
        }

        @Override
        public void run() {
            System.out.println(num1 * num2);
        }
    }

}

5、sleep方法

/**
* Causes the currently executing thread to sleep (temporarily cease
* execution) for the specified number of milliseconds, subject to
* the precision and accuracy of system timers and schedulers. The thread
* does not lose ownership of any monitors.
* 使当前执行的线程在指定的毫秒数内休眠（暂时停止执行）
* 这取决于系统计时器和调度程序的精度和准确性
* 并不会失去对任何监视器(monitors)的所有权
* 也就是说不会释放锁，仅仅会让出cpu的执行权
*/
public static native void sleep(long millis) throws InterruptedException;

  该方法可以使线程挂起指定的一段时间，比如Thread.sleep(1000)，表示将线程挂起1秒钟。在1秒钟之内放弃对CPU的竞争，挂起时间结束后重新竞争。由于CPU的时间片算法，一秒钟后该线程并不会被立即执行，必须等待CPU分配时间片。

  可以通过Thread.sleep(0)触发操作系统立刻重新进行一次CPU竞争，竞争的结果可能仍然是当前线程获得CPU时间片，也许是其他线程获得CPU时间片。与yield()方法功能相同。

# 停止线程

1、interrupt中断

  通过interrupt()方法来中断线程，将线程的中断状态置为true，而不是停止线程。我们定义的Runnable子类，需要在任务中对中断状态进行响应（检测中断状态），才可以通过中断停止线程。

public class InterruptTest {

    public static void main(String[] args) throws InterruptedException {
        Thread thread = new Thread(new PrintDateTask());
        thread.start();

        Thread.sleep(20);
        // 中断子线程
        thread.interrupt();

        Thread.sleep(20);
        System.out.println("主线程结束");
    }

    private static class PrintDateTask implements Runnable {

        @Override
        public void run() {
            while (!Thread.currentThread().isInterrupted()) {
                System.out.println("当前时间：" + new Date());
            }
            System.out.println("任务被中断，提前结束");
        }

    }

}

2、中断状态

  （1）如果线程正在调用一些阻塞方法时，此时线程被中断，会直接响应（检测到）中断状态，抛出InterruptedException异常。比如Thread#sleep(long)、Object#wait()等。同时清除目标线程的中断标志位（置为false）。

  （2）如果线程是被I/O或者NIO中的Channel所阻塞，同样I/O操作会被中断或者返回特殊异常值，达到终止线程的目的。

  （3）如果线程不满足挂起、等待、阻塞或者I/O所阻塞等条件的话，那么会将线程的状态设为中断状态。

public class InterruptTest {

    public static void main(String[] args) {

        Thread thread = new Thread(() -> {
            LockSupport.park();
            if (Thread.interrupted) {
                System.out.println("由于interrupt而被唤醒");
            } else {
                System.out.println("正常唤醒")
            }
        });

        // interrupt 使等待状态的线程唤醒
        thread.interrupt();

    }

}

3、废弃的stop()方式

  这种方式可能会导致线程的安全问题，造成数据不一致的情况。

public class UnsafeApiStopTest {

    private static int a = 100;
    private static int b = 100;

    public static void main(String[] args) throws InterruptedException {
        Thread thread = new Thread(() -> {
            a = a + 100;
            try {
                Thread.sleep(1000);
            } catch (InterruptedException e) {
            }
            b = b + 100;
        });
        thread.start();
        Thread.sleep(500);
        thread.stop();
        while (thread.isAlive()) {
        }
        // 对于a、b变量来说，要么全部成功，要么全部失败
        // 使用 stop的方式 造成了数据不一致的情况
        // 输出结果：   a：200,b：100
        System.out.println("a：" + a + ",b：" + b);
    }
}

# 生命周期

  （1）NEW，新生状态。创建了线程（new），但还未启动（start）。

  （2）RUNNABLE，运行状态。启动之后的线程状态，但需要等待操作系统的任务调度。

  （3）BLOCKED，阻塞状态。有三种情况，

  （3.1）当由于获取不到监视器锁，无法进入一个同步代码块或者同步方法时，就会处于阻塞状态。

  （3.2）进入了同步代码块或者同步方法后，在其中调用了wait()方法，释放了监视器锁。当线程被唤醒后，如果竞争不到监视器锁，同样会进入阻塞状态。

  （3.3）当JVM发生垃圾回收时，所有的工作线程都会处于阻塞状态。

  （4）WAITING。当调用Object#wait()、Thread#join()、LockSupport#park()时，就会进入等待状态。

  （5）TIMED_WAITING。当调用Thread#sleep(long)、Object#wait(long)、Thread#join(long)、LockSupport#parkNanos(long)或LockSupport#parkUntil(long)时，就会进入定时等待状态。

  （6）TERMINATED。线程正常或者异常结束，均会进入结束状态。

# 线程通信

  指多个线程在运行期间，线程之间的交互协作。

1、suspend与resume

  这是两个被废弃的API。suspend是挂起目标线程，但不会释放锁。resume来恢复目前线程。使用时必须保证suspend一定要比resume先执行，否则就无法被唤醒了，导致死锁的发生。

2、wait与notify

  方法wait()与方法notify()一定要在同步代码块中使用。虽然wait()会自动释放锁锁，但是对顺序有要求。如果notify()先于wait()使用，那么会造成死锁的发生。

public class NotifyTest {

    private static int[] numberArray = {1, 2, 3, 4, 5};
    private static char[] charArray = {'A', 'B', 'C', 'D', 'E'};
    private static final byte[] lock = new byte[0];
    private static CountDownLatch countDownLatch = new CountDownLatch(1);


    public static void main(String[] args) throws InterruptedException {

        Thread thread1 = new Thread(new PrintNumber());
        Thread thread2 = new Thread(new PrintChar());
        thread2.start();
        thread1.start();


    }

    private static class PrintNumber implements Runnable {

        @SneakyThrows
        @Override
        public void run() {
            // 保证优先打印数字
            countDownLatch.countDown();
            synchronized (lock) {
                for (int number : numberArray) {
                    System.out.println("数字：" + number);
                    // 唤醒其他线程
                    lock.notify();
                    // 阻塞当前线程
                    lock.wait();
                }
                lock.notify();
            }
        }
    }

    private static class PrintChar implements Runnable {

        @SneakyThrows
        @Override
        public void run() {
            countDownLatch.await();
            synchronized (lock) {
                for (char chars : charArray) {
                    System.out.println("字母：" + chars);
                    // 唤醒其他线程
                    lock.notify();
                    // 阻塞当前线程
                    lock.wait();
                }
            }
        }

    }

}

3、park与unpark

  线程调用park则等待许可，unpark方法为指定线程提供许可。不要求park和unpark方法的调用顺序。多次调用unpark之后，在调用park，线程会直接运行。但不会叠加，也就是说，连续多次调用park方法，第一次会拿到许可直接运行，后续调用会进入等待。park和unpark不会对锁进行释放。

public class NotifyTest {

    public class NotifyTest {

    private static int[] numberArray = {1, 2, 3, 4, 5};
    private static char[] charArray = {'A', 'B', 'C', 'D', 'E'};
    private static Thread thread1;
    private static Thread thread2;


    public static void main(String[] args) throws InterruptedException {

        thread1 = new Thread(new PrintNumber());
        thread2 = new Thread(new PrintChar());

        thread1.start();
        thread2.start();

    }

    private static class PrintNumber implements Runnable {

        @Override
        public void run() {
            for (int number : numberArray) {
                System.out.println("数字：" + number);
                LockSupport.unpark(thread2);
                LockSupport.park();
            }
        }

    }

    private static class PrintChar implements Runnable {

        @Override
        public void run() {
            for (char chars : charArray) {
                LockSupport.park();
                System.out.println("字母：" + chars);
                LockSupport.unpark(thread1);
            }
        }
    }

}

4、Condition

public class NotifyTest {

    private static int[] numberArray = {1, 2, 3, 4, 5};
    private static char[] charArray = {'A', 'B', 'C', 'D', 'E'};
    private static ReentrantLock reentrantLock = new ReentrantLock();
    private static Condition numberCondition = reentrantLock.newCondition();
    private static Condition charCondition = reentrantLock.newCondition();


    public static void main(String[] args) throws InterruptedException {

        Thread thread1 = new Thread(new PrintNumber());
        Thread thread2 = new Thread(new PrintChar());
        thread1.start();
        thread2.start();

    }

    private static class PrintNumber implements Runnable {

        @Override
        public void run() {
            reentrantLock.lock();
            try {
                for (int number : numberArray) {
                    System.out.println("数字：" + number);
                    charCondition.signal();
                    numberCondition.await();
                }
                charCondition.signal();
            } catch (InterruptedException e) {
                e.printStackTrace();
            } finally {
                reentrantLock.unlock();
            }
        }

    }

    private static class PrintChar implements Runnable {

        @Override
        public void run() {
            reentrantLock.lock();
            try {
                for (char chars : charArray) {
                    System.out.println("字母：" + chars);
                    numberCondition.signal();
                    charCondition.await();
                }
                numberCondition.signal();
            } catch (InterruptedException e) {
                e.printStackTrace();
            } finally {
                reentrantLock.unlock();
            }
        }
    }
}

# 伪唤醒

  官方建议应该在循环中检查等待条件，原因是处于等待状态的线程可能会收到错误警报和伪唤醒。如果不在循环中检查等待条件，程序就会在没有满足结束条件的情况下退出。涉及到底层的CPU调度。不要使用if判断是否进入等待状态。

  伪唤醒是指线程并非因为notify、notifyAll、unpark等api调用而唤醒，是更底层原因导致的。

synchronized(obj) {
    while(条件) {
        obj.wait();
    }
}

while(条件) {
    LockSupport.park();
}
