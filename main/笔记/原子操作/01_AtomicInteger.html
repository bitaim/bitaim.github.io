# AtomicInteger

public class CASTest {

    private volatile int value;

    private static Unsafe unsafe;

    private static long valueOffset;

    static {
        // 通过反射获取Unsafe
        Field field = Unsafe.class.getDeclaredField("theUnsafe");
        field.setAccessible(true);
        unsafe = (Unsafe) field.get(null);
        // 获取number属性偏移量（用于定位value属性在内存中的具体地址）
        valueOffset = unsafe.objectFieldOffset(CASTest.class.getDeclaredField("value"));
    }

    public void add() {
        // CAS + 循环重试
        int current;
        do {
            // 每次获取最新值
            current = unsafe.getIntVolatile(this, valueOffset);
        } while(!unsafe.compareAndSwepInt(this, valueOffset, current, current + 1))
        // public final native boolean compareAndSwapInt(Object var1, long var2, int var4, int var5);
    }
}

  Compare And Swap（Compare And Exchange）比较和交换，被称为自旋锁， 也被称为无锁。CAS是一种乐观锁。

  当多个线程修改同一个变量时，当采用加锁的方式时，是保证同一时刻只有一个线程能够执行。 而CAS无锁讲究的是，多个线程可以同时修改，但只有一个线程可以修改成功，而修改失败的线程则继续重试，这个过程是依赖硬件来实现的。

  CAS的实现主要调用的Unsafe类的CAS方法，这些都是native方法。CAS在底层有一条对应的汇编指令，硬件级别直接支持。native底层方法，具体实现在unsafe.cpp中。

  通过lock指令和cmpxchg指令（比较并交换）来实现底层的CAS操作。只靠cmpxchg指令是达不到CAS的，万一在比较之中，其他CPU修改了变量怎么办，cmpxchg指令不是原子性的。所以需要lock指令，在硬件级别上当执行cmpxchg指令时，其他CPU不允许做修改。

3、CAS的优势

  CAS是非阻塞的，不会产生死锁问题。线程间的相互影响也比较小，使用无锁的方式完全没有锁竞争带来的系统开销，也没有线程间频繁调度带来的开销。比基于锁的实现方式有更优秀的性能。

4、CAS的缺点

  循环时间太长。如果CAS长时间不成功，就会持续循环、自旋，必然给CPU带来非常大的开销。在JUC的有些地方就限制了CAS的自旋次数，例如BlockingQueue的SynchronousQueue

  只能保证一个共享变量原子操作。
