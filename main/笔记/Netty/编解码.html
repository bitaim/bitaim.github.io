# 编解码

  在网络编程中，需要解决编解码的问题。

  比如粘包问题，客户端发送了两条消息ABC和DEF。而服务端收到的消息可能是ABCDEF，两条消息变成了一条消息。

  比如半包问题，客户端发送了两条消息ABC和DEF。而服务端收到的消息可能是AB、CD、EF三条消息。

  在收发角度来看，一次发送可能被多次接收，多次发送可能被一次接收。在传输角度来看，一次发送可能占用多个传输包，多次发送可能共用一个传输包。

1、为什么TCP应用中会出现粘包问题？

  （1）客户端每次发送消息比较小，远小于套接字缓冲区大小。可能会出现合并发送的情况，这样能提高效率，但会造成服务端粘包的问题。

  （2）服务端接收套接字缓冲区的消息不及时，而导致粘包问题。

2、为什么TCP应用中会出现半包问题？

  （1）客户端发送消息大于套接字缓冲区大小。

  （2）客户端发送的消息大于协议的MTU最大传输单元，必须拆包。

3、根本原因

  TCP是流式协议，消息无边界。

  UDP像邮寄的包裹一样，虽然一次性传输多个，但包裹之间的界限清晰，一个一个接收即可。不会出现粘包、半包问题。

# 解码器

  解决问题就是要找到消息的边界。

  Netty相对于NIO的优势，在于Netty中自带了常用的编解码器。比如基于固定长度的解码器，FixedLengthFrameDecoder。基于分隔符的解码器，DelimiterBasedFrameDecoder。基于长度字段的解码器，LengthFieldBasedFrameDecoder。这三种解码器均继承自ByteToMessageDecoder。

1、优劣势

  固定长度解码器。满足固定长度即可，使用简单。但却会造成空间浪费，另外长度固定多少合适，又很难抉择。在生产环境中几乎没人使用。

  分隔符解码器。通过特殊字符进行分割，使用同样简单。但内容本身出现分隔符时需要转移。

  长度字段解码器。基于长度字段的解码器，先解析固定长度字节，通过长度获取后续内容，可以精准定位用户消息，内容也不需要转移。长度理论上有限制，需要提前预知最大长度。强烈推荐这种方式，同样是大量开源框架的首选方式，比如dubbo、rocketMQ、spark等。

# 长度字段解码器

import io.netty.handler.codec.LengthFieldBasedFrameDecoder;

/**
 * 基于长度字段解码器，构建符合自身业务的解码器
 */
public class FrameDecoder extends LengthFieldBasedFrameDecoder {

    public FrameDecoder() {
        super(128, 4, 4, 0, 8);
    }

}

LengthFieldBasedFrameDecoder构造方法中主要由5个参数，分别如下。

maxFrameLength：单个消息的最大长度，要根据自身业务场景而定

lengthFieldOffset：长度字段在整个消息中的偏移量

lengthFieldLength：长度字段所占的字节数，比如int型占4位，long型占8位

lengthAdjustment：在大多数情况下，长度字段仅表示消息正文的长度。但是，在某些协议中，长度字段表示整个消息的长度，包括消息头。比如长度字段的值为18（4+14），长度字段所占字节数为4，消息正文的所占字节数为14。在这种情况下，我们指定lengthAdjustment为-4来进行补偿

initialBytesToStrip：要丢弃的字节数量，比如去掉魔数、长度所占的字节，只剩下消息正文

# 二次解码器

  一次解码器解决粘包、半包的问题，获取到界限分明的字节流。通过合适的序列化/反序列化方案，完成字节数组和Java对象的转换，方面于后续处理器执行业务。

public class ObjectDecoder extends MessageToMessageDecoder<ByteBuf> {

    @Override
    protected void decode(ChannelHandlerContext ctx, ByteBuf msg, List<Object> out) throws Exception {
        // 使用JSON转换成Java对象
        String json = msg.toString(CharsetUtil.UTF_8);
        Request request = JSONObject.parseObject(json, Request.class);
        out.add(request);
    }

}

1、序列化方案选择

  （1）JSON。可读性好，占用空间小。推荐。

  （2）Protobuf。性能非常好，跨语言。但可读性最差。

  （3）Java序列化。占用空间比较大，只有Java语言能用，很难与其他语言做继承。几乎没人用。

# 源码

1、ByteToMessageDecoder

  这三种解码器都继承自ByteToMessageDecoder抽象类，而ByteToMessageDecoder又继承自ChannelInboundHandlerAdapter。 ChannelInboundHandlerAdapter的核心方法是处理数据，也就是channelRead方法。ByteToMessageDecoder的具体源码实现如下。

public abstract class ByteToMessageDecoder extends ChannelInboundHandlerAdapter {

    @Override
    public void channelRead(ChannelHandlerContext ctx, Object msg) throws Exception {
        // 接收到的msg可能是粘包或者半包的msg
        if (msg instanceof ByteBuf) {
            RecyclableArrayList out = RecyclableArrayList.newInstance();
            try {
                ByteBuf data = (ByteBuf) msg;
                // cumulation 是数据积累器
                first = cumulation == null;
                // 如果是第一笔数据的话
                if (first) {
                    // 直接将 数据 放入数据积累器里面
                    cumulation = data;
                } else {
                    // 不是第一笔 就追加到数据积累器里面 
                    cumulation = cumulator.cumulate(ctx.alloc(), cumulation, data);
                }
                // 调用解码器
                callDecode(ctx, cumulation, out);
            } catch (DecoderException e) {
                throw e;
            } catch (Throwable t) {
                throw new DecoderException(t);
            } finally {
                if (cumulation != null && !cumulation.isReadable()) {
                    cumulation.release();
                    cumulation = null;
                }
                int size = out.size();
                decodeWasNull = size == 0;

                for (int i = 0; i < size; i ++) {
                    ctx.fireChannelRead(out.get(i));
                }
                out.recycle();
            }
        } else {
            ctx.fireChannelRead(msg);
        }
    }

    protected void callDecode(ChannelHandlerContext ctx, ByteBuf in, List<Object> out) {
        try {
            // in 是数据积累器的数据
            while (in.isReadable()) {
                // 代码有删减
                // 调用具体子类的解码实现
                decode(ctx, in, out);
            }
        }
    }
}

2、三种解码器实现

  类FixedLengthFrameDecoder的解码实现。

public class FixedLengthFrameDecoder extends ByteToMessageDecoder {

    // 帧固定长度的大小
    private final int frameLength;

    public FixedLengthFrameDecoder(int frameLength) {
        if (frameLength <= 0) {
            throw new IllegalArgumentException(
                    "frameLength must be a positive integer: " + frameLength);
        }
        this.frameLength = frameLength;
    }

    @Override
    protected final void decode(ChannelHandlerContext ctx, ByteBuf in, List<Object> out) throws Exception {
        Object decoded = decode(ctx, in);
        if (decoded != null) {
            out.add(decoded);
        }
    }


    // 具体解码实现
    protected Object decode(
            @SuppressWarnings("UnusedParameters") ChannelHandlerContext ctx, ByteBuf in) throws Exception {
        // 如果in的数据长度 小于 设置的固定长度
        if (in.readableBytes() < frameLength) {
            // 那就解不出来数据
            return null;
        } else {
            // 满足解出来的情况 也有可能粘包了 粘包的多出来的数据还在数据积累器里面
            // 每调用retain方法一次，引用计数器就会加一。
            return in.readSlice(frameLength).retain();
        }
    }
}
