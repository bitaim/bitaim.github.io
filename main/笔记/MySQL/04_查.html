# 索引

  在开发中，通常会建立索引来提升SQL的执行效率，同时也能减少服务器的压力。

  MySQL的索引有以下几种，分别是主键索引、普通索引、唯一索引、联合索引、前缀索引。

# 1、索引的一些命令

  查看索引，SHOW INDEX FROM tb_user

  删除索引，DROP INDEX idx_username ON tb_user

# 2、索引的合理使用

  （1）避免将索引建立在区分度不大的字段上。比如描述状态的字段。

# 3、索引失效的场景

  （1）使用 != 或 <>（同 !=） ，放弃索引。

  （2）使用 OR 来连接条件，放弃索引。可使用 union all。

  （3）使用 %like 时，放弃索引。

  （4）使用函数或进行列计算，放弃索引。

  （5）隐式类型转换，隐式字符编码，放弃索引。

# 4、其他优化

  （1）如果确认查询结果是一条，建立在SQL语句后添加LIMIT 1

  （2）尽量不要使用SELECT *，应具体到字段。可以节省资源、减少网络开销。SELECT *很可能不会使用到覆盖索引了。

  （3）三种外关联JOIN，优先使用INNER JOIN。如果使用LEFT JOIN，要保证小表驱动大表。

  （4）优化深度分页，使用select * from test a inner join (select id from test where val=4 limit 300000,5)

  （5）慎用DISTINCT关键字，会占用很多的系统资源。

  （6）避免在列上使用函数。

-- 优化前
where date_format(create_time，'%Y-%m-%d') = '2019-07-01';
-- 优化后
where create_time between '2019-07-01 00:00:00' and '2019-07-01 23:59:59';

  （7）避免做复杂的表关联查询，既不利于维护和理解，又不利于复用。互联网的代码风格，95%的SQL都是基于单表的增删改查，join逻辑在代码里做。优点是SQL不需要做复杂调优，配合InnoDB的聚簇索引和几个组合索引，就可以覆盖所有的业务逻辑。另外SQL越简单，后续做分库分表、读写分离就会越简单，成本越低。

# 主键索引

  在创建表时，都会创建一个主键列。索引根据是否为主键，可以分为两大派，一派是主键索引（聚簇索引），另一派是非主键索引。

  因为在数据存储上有很大的区别。主键索引的叶子节点存储的是完整的行数据。非主键索引的叶子节点存储的是主键ID的值。

  基于非主键索引的查询，需要额外扫描一棵索引树（主键索引）。首先通过非主键索引定位到叶子节点上的主键ID，再通过主键索引定位到具体的行数据，这个过程叫做回表。

  使用覆盖索引来减少回表次数，是一个常用的性能优化手段。比如根据手机号查询主键，由于叶子节点存储的就是主键ID的值，因此不需要到主键索引所在的索引树上查询数据，这就是减少回表。通常我们会通过组合索引来使用到覆盖索引。

# 普通索引

  创建普通索引，CREATE INDEX idx_username ON tb_user(username)

  使用普通索引查询数据时，当查找到满足条件的第一个记录后，会继续向后查找记录，直到碰到第一个不满足条件的记录后，就会停止向后检索。

  InnoDB的读写数据是以数据页为单位的，普通索引查找到第一个记录后，就会将对应的数据页加载到内存（buffer pool）中。一般向后检索的时候就是基于内存在检索了，只需要额外的指针寻址和计算而已。

# 唯一索引

  创建唯一索引，CREATE UNIQUE INDEX uk_card ON tb_user(card)

  使用唯一索引查询数据时，查找到第一个满足条件的记录后，就会停止向后检索。

# 联合索引

  创建联合索引，CREATE INDEX idx_username_age ON tb_user(username,age)

1、最左前缀原则

  当创建联合索引时，根据索引的复用能力安排字段的先后顺序。比如创建一个（username,age）的组合索引，根据最左前缀原因，相当于创建了两个索引（username）和username,age。既满足了根据用户名和年龄查询用户的需求，又满足了根据用户名查询用户的需求。

  通常会使用覆盖索引来提升SQL的执行效率，但是索引字段的维护也是需要代价的，通过建立冗余索引来支持覆盖索引就需要权衡考虑了。

2、索引下推

  MySQL在5.6版本引入了索引下推优化。在遍历过程中，基于当前索引中包含的字段先进行判断，直接过滤掉不满足条件的数据，减少回表次数。

# 前缀索引

  邮箱这类的字段适合使用前缀索引，前缀索引占用的空间会更小，这就是前缀索引的优势。但同时带来的损失是，可能会增加额外的记录扫描次数，建立索引时关注的是区分度，区分度越高越好。另外，使用前缀索引就无法使用覆盖索引对查询性能的优化了。
